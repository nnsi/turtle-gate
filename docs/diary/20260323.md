# 2026-03-23 sparse-mode NaN バグ修正 — パイプラインが実は壊れていた

## やったこと

B-6ペーパートレード Day 2。ユーザーから「今日の6時のログ取れてる？」と聞かれて確認したところ、パイプラインは動いていたがシグナルが0件だった。調査の結果、3つのバグ＋1つの根本原因を発見・修正した。

1. **根本原因: sparse-mode NaN問題**
   - `buildReturnMatrix(sparse=true)` が日米祝日差でNaN行を生成
   - `estimateCfull` が「1行でもNaN→unclean」判定 → 全28ティッカーが弾かれて 0/28 clean
   - `generateSignalForDate` もwindow内の祝日NaN行で全ティッカーを弾く → 0 signals
   - 修正: clean判定を≥50%有効に緩和、window内NaN行をフィルタ、wider windowで補填
   - 結果: 0/28→26/28 clean, 0→3784 signal dates

2. **daily-report undefined**: `i.symbol`→`i.label`, `i.changePercent`→`i.changePct*100`
3. **review出力重複**: Claude Write tool + tee の二重書き込み。Write指示を削除
4. **Frobenius NaN**: drift monitorも同じsparse NaN問題。共通取引日フィルタ追加で0.1719に復旧

7時のClaude自動レビューも正常稼働を確認。全修正をコミット・プッシュし、翌日のパイプラインで正常動作を確認した（3/24 Day 3: 26/28 clean, 3784 signals, Frobenius 0.17）。

## 感想

### 3/22のレビュー9ラウンドでこのバグを見つけられなかったのが痛い

3/22の日記に「Cfull推定時のNaN汚染（correlationMatrixに疎行列を渡していた）」と書いている。同じ日にCfullのNaN問題を修正したはずなのに、sparse modeとの相互作用は見落としていた。CodexもClaudeも「関数単位」でレビューしていたので、`buildReturnMatrix(sparse=true)` → `estimateCfull` → `generateSignalForDate` を跨ぐデータフローの問題に気づけなかった。

結局、本番環境（タスクスケジューラ経由の自動実行）で初めて顕在化した。テスト時はCSVデータ（非sparse）を使っていたか、または短期データで祝日が少なかったのだと思う。「関数単位のレビュー」と「E2Eのデータフローテスト」は別物という教訓。

### 0/28→26/28の推論過程

サブエージェントに調査を投げたら「XLCとXLREが2010-2014に存在しなかったから」という回答が返ってきた。しかし2銘柄だけでは0/28にならない。ここで自分で考え直して「日米祝日差で全ティッカーにNaNが入る」という本当の原因に辿り着いた。

教訓: サブエージェントの調査結果は「ヒント」であって「答え」ではない。特に「なぜ0/28なのか」のような定量的な整合性チェックは自分でやる必要がある。2/28と0/28は全く違う原因を示唆する。

### signalsが0→3784に復旧した後も最初はまだ0だった

`estimateCfull`を直して26/28 cleanになったが、`generateSignalForDate`のwindow内NaN問題が残っていてまだ0 signals。2段階目のバグに気づくのに1回のテスト実行が必要だった。

さらに、window行のNaN filterを入れた後もまだ0だった。window size L+1=61行から祝日5-7行が消えて54-56行になり、`vWindow.length < params.L + 1` チェックに引っかかっていた。3段階目。

こういう「直したはずなのにまだ動かない」ループは精神的にきついが、毎回テスト実行→ログ確認→次のボトルネック特定というサイクルを回せたのは良かった。

## 技術メモ

- sparse mode（`buildReturnMatrix(..., true)`）は§8.1.2のXLC/XLRE動的ユニバース縮小のために導入されたが、副作用として全ティッカーに祝日NaNが入る。Cfull推定とwindow演算の両方で共通取引日フィルタが必要
- `estimateCfull`のclean判定: 0%→50%閾値に変更。XLC/XLREは2010-2014期間に存在しないので50%未満、他は祝日程度なので50%超
- `generateSignalForDate`: wider window（L*1.25行）を渡し、NaN行をフィルタ後に最新L+1有効行を使う
- `changePct`はdecimal格納（-0.0207 = -2.07%）。表示時に×100が必要
